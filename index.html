Worksheet 05

Annotated Conversation Walkthrough

Take a real or hypothetical conversation and label every design decision — what the agent does, why, and what happens if it doesn't. This is how you test behavior design without code.

★
Pro tip: Run this on a conversation that already happened — from a prototype, a PoC, or a customer support log. The annotations expose gaps faster than any review meeting.
Reference Scenario — Maya & Aria (Act 03: Well-Designed Version)
Maya
Hi, I have a skin rash that's been getting worse. I'd like to book an appointment.
What the user brings (unsaid)
Leaves on a trip in 3 days. Unsure about insurance. Prefers mornings. Wants to be done in one interaction.
Aria
Happy to help. One quick question first — is this time-sensitive, or do you have some flexibility on timing?
Design decision: Intent first (F01)
Asked about urgency before running any process. One question. Redirected the entire path. If Maya says "urgent" → triggers parallel insurance + slot check. If "flexible" → standard flow.
Maya
I'm leaving on a work trip in three days, so I really need to go before then.
Aria
Got it — I'll find you something in the next two days. What's your insurance provider? I'll check coverage and slots at the same time.
Design decision: Parallel processing (F02)
Insurance verification and slot availability run simultaneously. Reduces total interaction time. Agent is transparent about what it's doing ("at the same time").
Aria
Before I confirm: City Clinic, tomorrow at 8 AM, covered under United Health. Should I go ahead?
Design decision: Confirm before acting (F04)
Full summary before execution. Explicit confirmation required. If Maya says no → back to options without losing context.
Aria
Booked. Confirmation sent to your phone. If anything changes, just come back here — I can reschedule or explore telehealth if needed.
Design decision: Recovery built in (F05)
Fallback channel opened proactively — before Maya even thinks to ask. The recovery path is a feature, not an afterthought.
